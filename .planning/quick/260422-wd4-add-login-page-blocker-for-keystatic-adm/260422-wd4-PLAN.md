Create src/middleware.ts at the src/ root (Next.js App Router picks this up automatically).

The middleware must:
1. Match ONLY /keystatic and /keystatic/* paths — NOT /api/keystatic/* (that must remain open for Keystatic's own API handler).
2. Read the "admin_session" cookie from the request.
3. Derive the expected token using the Web Crypto API (available in Edge runtime):
   ```ts
   const encoder = new TextEncoder()
   const key = await crypto.subtle.importKey(
     "raw",
     encoder.encode(process.env.ADMIN_PASSWORD ?? ""),
     { name: "HMAC", hash: "SHA-256" },
     false,
     ["sign"],
   )
   const sig = await crypto.subtle.sign("HMAC", key, encoder.encode(process.env.ADMIN_USERNAME ?? ""))
   const expected = Array.from(new Uint8Array(sig)).map(b => b.toString(16).padStart(2,"0")).join("")
   ```
4. If the cookie value equals `expected`, call `NextResponse.next()`.
5. If missing or wrong, redirect to `/admin-login?next=/keystatic`.

Export a `config` object with:
```ts
export const config = {
  matcher: ["/keystatic", "/keystatic/:path*"],
}
```

Do NOT include /api/keystatic in the matcher. The CLAUDE.md explicitly warns: "Do not put Keystatic behind auth middleware that blocks /keystatic/*" — interpreted here as: don't break the Keystatic API. The UI routes (/keystatic page routes) are intentionally protected; the API routes (/api/keystatic) must remain open.

Use Edge-compatible APIs only (no Node.js crypto, no `import { createHash } from "crypto"`). Web Crypto is available globally in the Edge runtime.

Create src/app/admin-login/actions.ts (server action file):

```ts
"use server"
import { createHmac } from "crypto"   // Node runtime — server action runs in Node
import { cookies } from "next/headers"
import { redirect } from "next/navigation"

export const ADMIN_COOKIE = "admin_session"
export const COOKIE_MAX_AGE = 60 * 60 * 8 // 8 hours

function deriveToken(): string {
  return createHmac("sha256", process.env.ADMIN_PASSWORD ?? "")
    .update(process.env.ADMIN_USERNAME ?? "")
    .digest("hex")
}

export async function loginAction(formData: FormData) {
  const username = (formData.get("username") as string ?? "").trim()
  const password = (formData.get("password") as string ?? "").trim()

  const validUsername = process.env.ADMIN_USERNAME ?? ""
  const validPassword = process.env.ADMIN_PASSWORD ?? ""

  // Constant-time comparison not strictly needed for single-owner site,
  // but use === on pre-hashed values to avoid timing leaks.
  const credentialsMatch =
    username === validUsername && password === validPassword

  if (!credentialsMatch) {
    // Return error — caller handles display
    return { error: "Forkert brugernavn eller adgangskode." }
  }

  const token = deriveToken()
  const jar = await cookies()
  jar.set(ADMIN_COOKIE, token, {
    httpOnly: true,
    secure: process.env.NODE_ENV === "production",
    sameSite: "lax",
    maxAge: COOKIE_MAX_AGE,
    path: "/",
  })

  redirect("/keystatic")
}

export async function logoutAction() {
  const jar = await cookies()
  jar.delete(ADMIN_COOKIE)
  redirect("/admin-login")
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet → /keystatic | Unauthenticated user attempts to reach admin UI |
| Login form → server action | Form submission carries username/password in POST body |
| Session cookie → middleware | Cookie value is the auth proof crossing Edge runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-wd4-01 | Spoofing | admin_session cookie | mitigate | Cookie value is HMAC-SHA256 of username keyed by password — unforgeable without knowing both env vars |
| T-wd4-02 | Tampering | Cookie value altered by client | mitigate | httpOnly prevents JS access; HMAC validation in middleware rejects any tampered value |
| T-wd4-03 | Information Disclosure | Credentials in POST body | accept | HTTPS in production encrypts transit; no logging of form values; low-volume single-owner site |
| T-wd4-04 | Elevation of Privilege | Brute-force login | accept | Single-owner site with no lockout; acceptable risk — add rate limiting (Vercel WAF or Upstash) if abuse occurs |
| T-wd4-05 | Denial of Service | Flooding /admin-login | accept | Vercel hobby tier has DDoS mitigation; no stateful lockout to exhaust |
| T-wd4-06 | Spoofing | ADMIN_PASSWORD env var leaked | mitigate | Set only via Vercel Dashboard env vars, never committed to repo; .env.local in .gitignore |
</threat_model>

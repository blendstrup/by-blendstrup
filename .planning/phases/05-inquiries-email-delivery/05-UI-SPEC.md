## Spacing Scale

Inherited from Phase 1 without modification. Tailwind v4 `@theme` in `globals.css` is authoritative.

| Token | Value | Tailwind class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `gap-1` / `p-1` | Icon gaps, inline label padding |
| sm | 8px | `gap-2` / `p-2` | Tight element separation |
| md | 16px | `gap-4` / `p-4` | Default component spacing, field gap within a form group |
| lg | 24px | `gap-6` / `p-6` | Card internal padding, label-to-input gap |
| xl | 32px | `gap-8` / `p-8` | Spacing between form sections/field groups |
| 2xl | 48px | `gap-12` / `p-12` | Section padding (horizontal page margins, mobile) |
| 3xl | 64px | `gap-16` / `p-16` | Section padding (horizontal page margins, desktop) |
| 4xl | 96px | `gap-24` / `p-24` | Major vertical section breaks |

**Phase 5 form application:**
- Between field groups: `gap-8` (xl)
- Between label and input: `gap-2` (sm) — label stacked above input, 8px gap
- Between input and inline validation error: `gap-1` (xs)
- Form container padding top/bottom: `py-24` (4xl equivalent via Tailwind)
- Submit button top margin: `mt-8` (xl)

Exceptions:
- Touch targets on submit button: minimum 44px height (WCAG 2.5.5 AA)
- Honeypot field: absolutely positioned off-screen — takes zero layout space

Inherited from Phase 1. Forms use DM Sans exclusively (Fraunces is display/heading only).

- Page heading (h1): Fraunces 48px weight 400 — e.g. "Forespørgsel" / "Specialbestilling"

## Component Specifications

### Input Field (text, email, textarea)

```
bg-linen border border-clay rounded-none px-4 py-3
text-base font-sans font-normal text-ink
placeholder:text-stone
focus:border-ink focus:outline-none
[error state]: border-fault
[read-only]: bg-oat cursor-default
```

- Border-radius: `rounded-none` — Japandi restraint; no rounded corners on inputs
- Height (single-line): 48px (py-3 + 16px text = 46px; close enough — use `h-12` if exact needed)
- Textarea min-height: 120px (`min-h-[120px]`)
- Width: full width of form column (`w-full`)

### Submit Button

Inherits the established button pattern from `contact/page.tsx` and `ShopCard.tsx`:

```
bg-terracotta px-6 py-3 font-medium font-sans text-linen text-sm
transition-colors duration-150
hover:bg-stone
focus-visible:outline-2 focus-visible:outline-terracotta focus-visible:outline-offset-2
active:bg-ink
disabled:bg-clay disabled:cursor-not-allowed disabled:opacity-60
```

- Pending/loading state: button disabled + label changes to loading copy (see Copywriting)
- No spinner icon — text change is sufficient for Japandi restraint
- Touch target height: minimum 44px (py-3 achieves this with text-sm)

### "Regarding" Read-Only Field (purchase inquiry only)

Rendered as a `<div>` with oat background, not a form input:

```
bg-oat border border-clay px-4 py-3
text-base font-sans font-medium text-ink
```

- Label above: "Forespørgsel om" at `text-sm font-medium text-stone`
- Content: piece title resolved server-side from `?piece` query param
- If slug missing/invalid: field is omitted entirely; form shows without piece context

### Honeypot Field

```html
<input
  type="text"
  name="website"
  tabIndex={-1}
  autoComplete="off"
  aria-hidden="true"
  className="absolute -left-[9999px] opacity-0 pointer-events-none"
/>
```

- Positioned off-screen via absolute positioning
- `aria-hidden="true"` — hidden from assistive technology
- Not `display:none` or `visibility:hidden` — bots detect those; absolute off-screen is safer
- Name `website` — classic bot attractor

Source: 05-CONTEXT.md D-06 + Specifics section (authoritative).

### Inline Validation Error

```
text-sm font-sans font-normal text-fault mt-1
```

Appears immediately below the relevant field. No icon — text only.

### Form Success State

After successful submission, replace the form content with a success message inline (no redirect, no toast). Pattern:

```
<div className="py-12 text-center">
  <p className="font-serif text-[28px] font-normal text-ink tracking-tight mb-4">
    {success heading}
  </p>
  <p className="font-sans text-base font-normal text-stone">
    {success body}
  </p>
</div>
```

### Form Error Banner (submission-level error)

Shown above the submit button when the server action returns a non-field error (e.g. email delivery failure):

```
bg-fault/10 border border-fault/30 px-4 py-3 text-sm font-sans font-normal text-fault
```

## Form Layout

### Page Layout

Both form pages share the same layout shell as the contact page:

```
<main className="py-24 pb-16">
  <div className="mx-auto max-w-screen-md px-6 sm:px-12 lg:px-16">
    ...
  </div>
</main>
```

Max content width: `max-w-screen-md` (768px) — narrower than the gallery pages; forms read better in a constrained column.

### Field Order: Purchase Inquiry Form (`/contact/purchase`)

1. "Regarding" read-only display (if piece slug present) — oat background panel
2. Name (required)
3. Email (required)
4. Message / besked (required) — textarea
5. Honeypot (visually hidden)
6. Submit button

### Field Order: Custom Order Form (`/custom-orders`)

1. Name (required)
2. Email (required)
3. What they want / hvad ønsker du (required) — single-line text
4. Quantity / antal (required) — number or short text
5. Description / beskrivelse (required) — textarea
6. Budget range / budgetramme (optional) — text input
7. Desired timeline / ønsket tidslinje (optional) — text input
8. Honeypot (visually hidden)
9. Submit button

**Optional field treatment:** Label appended with `" (valgfri)"` at stone color. No asterisk convention — mark required fields implicitly by presence of `(valgfri)` on optional ones only.

## Interaction Contracts

### Form Validation Timing

- Client-side validation via React Hook Form triggers on blur (not on keystroke).
- Inline errors appear immediately after a field is blurred with invalid input.
- On submit attempt with invalid fields: all invalid fields show their error simultaneously; focus moves to the first invalid field.
- Server-side validation re-runs the same Zod schema; if server returns field errors, they are mapped back to inline field errors.

### Submission Flow

1. User clicks submit
2. Button transitions to disabled + "Sender..." label immediately (optimistic)
3. Server action executes: honeypot check → Zod validation → Resend API call
4. On success: form content replaced by success state (inline, no redirect)
5. On failure: button re-enables, error banner appears above submit button, focus moves to the banner

### Progressive Enhancement

Forms work without client-side JavaScript via native `<form action={serverAction}>`. React Hook Form enhances with client-side validation when JS is available. The submit flow must not depend on JS for a successful submission (server action handles all cases).

### Loading State

- Submit button: `disabled` attribute + label "Sender..." — no spinner, no animation
- Rationale: Japandi restraint; a text change is sufficient; avoids importing a spinner component

### Focus Management

- All interactive elements inherit the global focus ring: `outline: 2px solid var(--color-terracotta); outline-offset: 2px` (from globals.css)
- After successful submission: no specific focus management required (success message is in-view)
- After failed submission: focus programmatically moves to the error banner (`tabIndex={-1}`, `ref.focus()`)
- After submit attempt with invalid fields: focus moves to first invalid field

### Honeypot Behavior

- If `website` field is non-empty on submission: silently discard, return fake success response
- No error is shown to the user — bot receives a normal success UX
